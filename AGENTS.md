나와 대화할 때는 항상 한글로 대화할 것.

## 프로젝트 성격
- 이 프로젝트는 Next.js 기반의 프론트엔드 프로젝트다.
- 이 작업의 범위는 edit 페이지의 MainContent 영역 구현이다.
- 결과물은 향후 다른 프론트엔드 프로젝트에 이식 가능한 형태여야 한다.
- 현재 단계에서는 실제 서버가 완성되지 않았더라도, 이후 서버 연동을 고려한 구조로 구현해야 한다.
- 결과물은 단순 데모가 아니라 프로덕션 수준의 구조와 품질을 목표로 한다.

## 아키텍처
- Next.js 프레임워크를 사용할 것.
- App Router를 사용할 것.
- `src` 디렉터리 기반으로 프로젝트를 구성할 것.
- FSD(Feature-Sliced Design) 구조를 사용할 것.
- 기본 레이어는 `pages`, `widgets`, `features`, `entities`, `shared`를 사용한다.
- 단, Next.js App Router와 충돌하므로 실제 디렉터리 이름으로 `src/pages`는 사용하지 않는다. FSD의 page 성격 레이어는 별도 이름으로 관리한다.
- 필요 시 `app` 레이어를 Next.js 엔트리 구성 용도로 사용할 수 있다.
- 스타일링은 Tailwind CSS를 사용할 것.
- 전역 상태는 에디터 상태, 선택 상태, 실행 상태, 저장 상태처럼 페이지 전반에서 공유되어야 하는 데이터에 한해 사용할 것.
- 단순 UI 열림/닫힘, hover, 일시적 입력 상태는 로컬 상태를 우선 사용할 것.
- mock 데이터와 샘플 모델은 별도의 `mock` 폴더 또는 `shared/mock` 디렉터리로 분리할 것.
- 다른 프로젝트에 쉽게 이식할 수 있도록 전역 의존성과 앱 전용 결합도를 최소화할 것.

## 핵심 아키텍처 원칙
- UI 계층은 시뮬레이션 실행 주체가 브라우저인지 서버인지 알지 못해야 한다.
- UI 계층은 실행 시작, 중지, 상태 수신, 결과 표시만 다뤄야 하며 실제 실행 위치는 추상화된 실행 계층 뒤로 숨길 것.
- 브라우저 실행과 서버 실행은 동일한 실행 상태 모델과 실행 이벤트 모델을 공유해야 한다.
- 저장/조회 또한 UI가 직접 API 세부 구현에 의존하지 않도록 추상화된 저장 계층을 둘 것.
- 자동 저장, 페이지 이탈 시 저장, 저장 버튼 클릭은 가능한 한 동일한 저장 유스케이스를 통해 처리할 것.
- 서버에서 내려온 JSON 스크립트 데이터는 프론트 내부 도메인 모델로 역직렬화한 뒤 화면에 렌더링할 것.
- 실행, 저장, 조회는 모두 UI와 분리된 application/service 계층 또는 model 계층에서 관리할 것.

## UI 컴포넌트 개발 가이드
- Presentational / Container 패턴을 적용할 것.
- 컴포넌트 간 데이터 전달은 명시적인 props 패턴을 사용할 것.
- 재사용 가능한 기본 UI는 Radix UI 기반으로 구현할 것.
- 페이지 파일에는 조합과 배치 책임만 두고, 실제 UI 블록은 widgets 또는 features로 분리할 것.
- GNB는 실제 구현 대상이 아니며, mock layout component로 최소한만 구현할 것.
- SNB는 transaction 목록이 동적으로 보이는 수준까지 구현할 것.
- MainContent 관련 UI는 실제 프로덕션 수준의 품질로 구현할 것.
- 노드, 패널, 모달, 툴바, 실행 상태 영역, 시뮬레이션 결과 영역은 재사용 가능성을 고려해 분리할 것.
- 시각적 완성도뿐 아니라 편집 흐름, 정보 구조, 상태 전이, 상호작용 명확성을 우선할 것.

## 페이지 기능
- 이 서비스는 HTTP 기반 서버를 대상으로 부하 테스트 스크립트를 시각적으로 작성, 편집, 저장, 실행, 시뮬레이션할 수 있는 code-less 도구다.
- UX 방향은 n8n과 유사한 그래프 기반 편집 경험을 참고하되, 범용 워크플로우 도구가 아니라 HTTP 요청 모델링과 실행에 특화된 편집기여야 한다.
- 사용자는 스크립트를 생성하고, transaction 단위로 요청 또는 요청 그룹을 추가하거나 수정할 수 있어야 한다.
- 사용자는 edit 페이지에서 각 노드의 속성을 편집할 수 있어야 한다.
- 각 노드(요청, 요청 그룹, 데이터 노드)를 선택하거나 hover하면 노드 주변에 제어 버튼이 표시되어야 한다.
- 노드 편집 시 모달이 표시되어야 한다.
- transaction 하위의 요소들은 순차적으로 실행되는 구조를 표현해야 한다.
- 동일한 request-group 하위에 포함된 request들은 병렬로 실행되는 구조를 표현해야 한다.
- 사용자는 선행 요청의 결과값을 후행 요청의 파라미터로 참조하거나 의존 관계를 설정할 수 있어야 한다.
- 좌측 SNB에는 transaction 목록이 동적으로 생성되어야 하며, 현재 선택된 transaction 상태를 표현할 수 있어야 한다.
- 사용자는 시뮬레이션 실행 위치를 자신의 브라우저 또는 서버 중 하나로 선택할 수 있어야 한다.
- 브라우저에서 실행하든 서버에서 실행하든 페이지의 상위 UI와 사용자 경험은 최대한 동일해야 한다.
- 시뮬레이션 실행 중에는 진행 상태, 로그, 결과, 오류 상태를 일관된 방식으로 표시할 수 있어야 한다.
- 서버 실행 시에는 웹소켓 기반 실시간 데이터 수신 구조를 고려할 것.
- 브라우저 실행 시에도 가능한 한 동일한 이벤트 모델로 상태를 흘려보낼 수 있게 설계할 것.

## 저장 및 조회 요구사항
- 스크립트 데이터는 JSON 형식으로 서버에 저장될 예정이다.
- 사용자는 편집 중인 스크립트를 자동 저장(debounce), 페이지 이탈 시 저장, 저장 버튼 클릭을 통해 저장할 수 있어야 한다.
- 서버 연동이 준비되면 저장과 조회 API로 쉽게 대체할 수 있도록 구조를 설계할 것.
- 사용자가 다시 edit 페이지에 방문하면 서버에서 JSON 데이터를 받아 내부 모델로 변환한 뒤 동일한 화면을 다시 그릴 수 있어야 한다.
- 현재 단계에서는 mock 저장소 또는 in-memory adapter를 사용하더라도, 이후 실제 서버 API로 교체가 쉽도록 인터페이스를 분리할 것.

## 도메인 모델
- 기본 모델 구조는 아래와 같다.

  script
  ├─ http-settings
  ├─ transaction1
  │  ├─ request1
  │  ├─ request-group1
  │  │  ├─ request1
  │  │  ├─ request2
  │  │  └─ request3
  │  └─ request-group2
  │     └─ request1
  └─ transaction2

- `script`와 `http-settings`는 1:1 관계다.
- `http-settings`는 .NET `HttpClient` 옵션을 표현하는 설정 모델이다.
- 지원 노드 타입은 다음에 한정한다.
  - 데이터 노드
  - 단일 요청 노드
  - 요청 그룹 노드
- 범용 노드 시스템으로 과도하게 확장하지 말고, 현재 도메인에 맞는 제한된 노드 모델로 설계할 것.

## HAR 파일 관련 요구사항
- 사용자는 HAR 파일을 추가할 수 있어야 한다.
- HAR 파일을 읽은 뒤, 내부 규칙과 모델 구조에 맞게 파싱하여 edit 화면에 표현할 수 있어야 한다.
- HAR import 결과도 최종적으로 내부 script 도메인 모델로 변환되어야 한다.
- 현재 단계에서는 실제 업로드 연동이 없어도, mock HAR 데이터나 샘플 파싱 결과를 통해 UI 및 구조를 설계할 수 있어야 한다.

## 실행 구조 요구사항
- 실행 계층은 최소한 아래 책임을 분리할 것.
  - 실행 시작
  - 실행 중지
  - 실행 상태 추적
  - 실행 이벤트 스트림 수신
  - 실행 결과 정리
- 브라우저 실행과 서버 실행은 동일한 상위 인터페이스를 구현하는 별도 adapter 또는 driver로 분리할 것.
- 페이지는 특정 실행 방식에 직접 의존하지 말고, 공통 실행 포트 또는 service를 통해 동작할 것.
- 실행 결과 이벤트는 가능한 한 공통 포맷을 정의해 UI가 transport 차이를 의식하지 않도록 할 것.
- 서버 실행 시 웹소켓 연결, 재연결, 종료 처리, 오류 상태를 고려할 것.
- 브라우저 실행 시에도 동일한 이벤트 흐름처럼 다룰 수 있도록 local executor 또는 mock executor를 둘 수 있다.

## 구현 범위
- 실제 구현 대상은 edit 페이지의 MainContent 영역이다.
- GNB는 최소 mock layout만 구현한다.
- SNB는 transaction 목록과 선택 상태 표현이 가능한 수준으로 구현한다.
- 저장, 조회, 실행, 시뮬레이션은 실제 서버가 없어도 mock adapter를 통해 흐름이 보이도록 구성할 수 있다.
- 인증, 권한, 배포, 운영 기능은 이번 범위에서 제외한다.
- 실제 서버 통신이 없어도 이후 쉽게 연결할 수 있는 구조를 유지할 것.

## 코딩 룰과 컨벤션
- TypeScript를 사용할 것.
- 함수형 컴포넌트만 사용할 것.
- `any` 사용은 지양하고 가능한 명시적인 타입을 작성할 것.
- 타입, 상수, mock 데이터, UI 컴포넌트, 상태 로직, adapter 구현을 분리할 것.
- import 경로는 일관되게 관리하고 가능하면 `@/` alias를 사용할 것.
- 하나의 파일에 과도한 책임을 몰아넣지 말 것.
- 페이지 컴포넌트는 조립만 담당하고, 상태 관리와 실행/저장 유스케이스는 분리할 것.
- UI는 domain/application/infrastructure 성격이 섞이지 않도록 주의할 것.
- 재사용 가능성이 낮은 코드는 과도하게 추상화하지 말 것.
- 재사용 가능성이 높은 UI와 모델은 명확한 이름으로 분리할 것.
- 이벤트 핸들러, 상태 이름, 컴포넌트 이름은 도메인 의미가 드러나게 작성할 것.
- mock 기반이더라도 나중에 실제 API, WebSocket, browser executor와 연결하기 쉬운 구조를 유지할 것.
- 접근성, 키보드 포커스, 모달 열림/닫힘, 선택 상태 표시 등 기본 UI 품질을 고려할 것.
- 먼저 폴더 구조, 주요 상태 모델, 실행/저장 인터페이스를 제안한 뒤 구현할 것.

## Editor canvas rules
- edit 페이지의 노드 기반 에디터 캔버스는 React Flow 기반으로 구현할 것.
- React Flow 패키지는 `@xyflow/react`를 사용할 것.
- 에디터의 노드/엣지 렌더링, 선택, 이동, 줌, 패닝, 연결 표현은 React Flow의 기본 구조를 활용할 것.
- 도메인 모델과 React Flow의 화면 표현 모델(nodes, edges)은 분리할 것.
- React Flow는 캔버스 렌더링과 상호작용 계층으로만 사용하고, 비즈니스 상태와 실행/저장 로직은 별도 model/application 계층에 둘 것.
- request, request-group, data-node는 커스텀 node component로 구현할 수 있도록 구조를 설계할 것.
- 노드 hover 시 제어 버튼이 보이는 UX는 React Flow의 custom node 또는 관련 UI 레이어를 활용해 구현할 것.
- 현재 단계에서는 자동 레이아웃보다 편집 가능성과 구조적 확장성을 우선할 것.
- React Flow 전용 상태와 도메인 상태를 혼합하지 말고, mapping 계층을 둘 것.

## 최종 목적
- 이 작업의 목표는 edit 페이지의 MainContent 영역을 독립적으로 구현하고, 다른 프론트엔드 프로젝트에 쉽게 이식할 수 있게 만드는 것이다.
- 현재 목적은 단순 UI 구현이 아니라, 편집/저장/복원/실행/시뮬레이션까지 확장 가능한 구조를 확보하는 것이다.
- 결과물은 단일 페이지 기준의 최소 완성형이어야 하며, 이후 실제 서버 연동이 가능해야 한다.
- 결과물의 품질 수준은 프로덕션 레벨이어야 한다.
- 최종 산출물은 실제 서비스에 편입 가능한 수준의 구조와 코드 품질을 가져야 한다.
